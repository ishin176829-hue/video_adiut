## 1. 结论

当前接口功能和基础设施链路可用，但现有单机配置尚不满足“4-6 小时审核 10000 条分集视频”的生产容量要求。

| 验收项 | 结果 | 结论 |
| --- | ---: | --- |
| 自动化回归 | 86/86 通过 | 通过 |
| PostgreSQL / Redis / Redis 队列健康检查 | 均正常 | 通过 |
| 20 并发真实审核 | 20/20 完成 | 功能通过 |
| 80 并发真实审核 | 77 完成、3 失败 | 不通过 |
| 120 并发真实审核复测 | 100 完成、20 失败 | 不通过，源视频直拉连接超时先成为瓶颈 |
| 120 档峰值模型调用 | 约 103 QPM | 未达到 500 QPM |
| 模型 API 直压 500 QPM | 500/500 成功 | 通过，仅代表轻量文本/结构化文本模型调用 |
| 10000 条 / 4-6 小时容量 | 需要 1667-2500 条/小时 | 当前未达到 |
| 160/200/240/300 并发 | 未执行 | 120 档未过门槛，继续加压没有验收价值 |

当前建议：先把视频源改为 OSS 内网拉取或预热缓存，并把下载阶段与模型活跃槽拆开；120 档满足成功率不低于 99%、429 低于 5%、超时低于 3% 后，再决定是否继续测试更高档位。

### 5.1 模型 API 500 QPM 直压

2026-07-10 00:04（北京时间）在 `10.0.0.10` 上绕过下载、抽帧、Redis、PostgreSQL 和业务 worker，直接调用当前模型网关：

| 项目 | 值 |
| --- | --- |
| Base URL | `https://jzapi.duanju.com` |
| 模型 | `gemini-3.1-pro-preview` |
| 请求类型 | 结构化 JSON 文本请求 |
| 目标 QPM | 500 |
| 请求数 | 500 |
| 滑动 60 秒发起峰值 | 500 |
| 滑动 60 秒完成峰值 | 495 |
| 成功数 | 500 |
| 429 / 504 / timeout | 0 / 0 / 0 |
| 平均延迟 | 5.52s |
| P50 / P95 / P99 | 5.20s / 7.71s / 12.40s |

结论：当前模型网关对轻量文本和结构化 JSON 文本请求可以承受 500 QPM。该结论不能直接等价为完整视频审核 500 QPM，因为视频审核还包含源视频下载、FFmpeg 抽帧、OCR、4x4 拼图上传、结构化审核提示词、结果入库和回调；其中 120 并发复测已经证明源视频直拉阶段会先失败。

补充发现：`gemini-3.1-pro-preview` 在 JSON mode 下如果把 `max_output_tokens` 限到 64，会大量出现 `MAX_TOKENS` 空响应。生产审核链路不应给该模型设置过低输出上限。

## 6. 容量判断

10000 条任务的目标吞吐：

| 完成窗口 | 最低吞吐 |
| ---: | ---: |
| 6 小时 | 1666.7 条/小时 |
| 5 小时 | 2000 条/小时 |
| 4 小时 | 2500 条/小时 |

以 120 复测脚本终态吞吐约 273.2 条/小时估算，仍有约 6.1-9.2 倍差距。单纯把全局并发从 120 提升到 160 只会让更多任务堆在下载和模型尾延迟上，不能解决容量缺口，反而可能降低有效吞吐。

达到目标至少需要同时完成：

1. 与模型服务商确认稳定 QPS、并发和日配额，而不是只提升本机 Worker。
2. 降低每条视频模型请求数，继续优化帧去重、拼图和字幕通道合并策略。
3. OSS 内网下载、预处理和模型调用形成分阶段并发，避免下载占用审核活跃槽。
4. 按模型限流反馈做全局自适应并发，不使用固定并发持续撞 429。
5. 多实例水平扩容 Worker，并确保 Redis 全局槽和 PostgreSQL 连接池按总实例数核算。
6. 启用 OSS 上传链路，把审核输入改为 OSS object key；worker 从 OSS 内网 Endpoint 拉取，避免公网源 URL 成为首段瓶颈。

## 7. 压测发现与修复

| 问题 | 根因 | 修复状态 |
| --- | --- | --- |
| Worker 实际并发低于配置 | 每轮只拉取有限消息，任务池未持续补满 | 已修复并通过自动化测试 |
| 504 后递归拆分导致请求放大 | 所有瞬时错误共用拆分逻辑 | 已限制最大拆分深度 |
| 429 时拆分帧批次进一步放大 QPS | 未单独识别 rate limit | 已改为同批次指数退避，自动化测试通过；待 120 档真实复测 |
| 字幕通道超时导致整条视频失败 | 字幕异常直接向上抛出 | 已降级为继续视觉审核 |
| API 本地状态完成但 DB 仍处理中 | 终态持久化异常被吞掉，查询只看本地状态 | 已增加终态持久化重试和 DB 权威状态查询；待真实复测 |
| URL 下载前 HEAD + 下载后再次读盘计算哈希 | 多一次网络往返和一次完整磁盘读取 | 已取消 HEAD，下载流中增量计算 SHA256 |
| 下载无限等待占用活跃槽 | 只有连接超时，没有总超时 | 已增加总下载超时和进程内下载并发限制 |
| 压测产物推高磁盘占用 | 原视频、抽帧和报告未及时清理 | 已增加磁盘水位保护和按压测前缀清理 |
| 多 worker 同时启动偶发少进程 | 启动时并发执行 stale processing 回收，PostgreSQL 更新与事件插入出现 deadlock | 已加 PostgreSQL advisory lock；远端 86/86 通过，重启验证 10 个 worker 全部存活 |

## 8. 未通过项和剩余风险

1. 120 档真实复测已执行但未通过：120 条中 20 条 `ConnectTimeout`，源视频直拉阶段不稳定。
2. HMAC 只完成自动化正向用例；签名错误、时间戳过期、nonce 重放尚未在生产开启鉴权后做黑盒验证。
3. callback 已有签名单测，但尚未与真实交片系统接收端完成网络级联调；当前回调失败不自动重试。
4. 80 档有 3 个真实任务失败，120 复测有 20 个真实任务失败，均不满足成功率不低于 99% 的验收门槛。
5. 当前压测样本数量不足以证明 4-6 小时持续运行稳定性；通过短时档位测试后还需要至少 2 小时稳态 soak test。

## 9. 下一轮验收门槛

下一轮仍先复测 120 档，不直接测试 160：

| 指标 | 门槛 |
| --- | ---: |
| API / PostgreSQL 终态一致率 | 100% |
| 最终成功率 | 不低于 99% |
| 模型 429 比例 | 低于 5% |
| 模型 timeout 比例 | 低于 3% |
| 压测后残留 processing | 0 |
| 压测产物清理 | 100% |

120 档通过后，才测试更高档位；若 120 档仍有明显 429，应先下调并发或引入全局自适应限流。

120 档再次失败时，不再继续提高并发；先改造下载链路为 OSS 内网拉取、下载队列和审核队列分离，再重新压测。

# 模型审核工作流可靠性设计

## 目标

- 提交后 30 分钟内必须产出完整机审结果，否则计为最终技术失败。
- 每个滚动 10,000 个任务中，最终技术失败不得超过 1 个。
- `manual_review` 不能作为模型限流、熔断、超时或结构化输出失败的技术兜底。
- 429、临时网关错误、模型熔断和结构化输出错误必须在服务内部恢复，恢复期间任务保持非终态。

## 已确认根因

1. 结构化输出的 `parse` / `validation` 错误被计入全局供应商健康度，错误地触发全局熔断。
2. 熔断打开后没有完整的半开探测状态，恢复依赖 Redis 状态过期，容易形成长时间阻塞。
3. 帧批次在 429、解析或临时错误重试耗尽后生成“人工复核”伪结果，不符合完整机审要求。
4. 模型阶段没有 Redis 持久化延迟重试；任务异常后可能进入失败或死信。
5. worker 未处理的异常直接进入死信，未利用 Redis Streams pending 消息进行恢复。

## 状态机

工作流沿用现有 Redis Streams 分段队列和 PostgreSQL `review_jobs` / `review_events`，不引入新的调度框架。

```text
pending
  -> preprocess
  -> model_queued
  -> scan
  -> model_retry_wait
  -> scan (重试恢复，成功批次使用缓存)
  -> judge
  -> done

任一模型可恢复错误 -> model_retry_wait
30 分钟截止时间到达 -> failed
明确不可恢复错误 -> failed
```

每次延迟重试把以下状态写入请求 `metadata` 并写入 `review_events`：

- `workflow_started_at`
- `workflow_deadline_at`
- `model_retry_attempt`
- `model_retry_reason`
- `model_retry_error_kind`

Redis 有序集合保存待重试模型任务，到期后原子移动到模型 Stream。已完成帧批次继续使用现有帧缓存，PostgreSQL 分段结果继续使用 upsert，因此整阶段重跑保持幂等。

## 错误分类

### 参与全局熔断

- `rate_limit`: 429 / RESOURCE_EXHAUSTED
- `transient`: 502、503、504、连接重置、超时

### 不参与全局熔断

- `parse` / `validation`: 单个 Key 的输出质量问题，由 Key 池降权和跨 Key 重试处理
- `auth`: 单个 Key 禁用
- `permanent`: 请求或配置错误，直接终止
- `circuit`: 本地保护状态，不得再次记作上游错误

熔断状态为 `closed -> degraded -> open -> half_open -> closed/open`。打开期间不得被后续在途请求延长；到期后只允许一个探测调用，探测成功清空旧健康窗口并关闭，探测失败重新打开。

## 重试层级

1. **调用级重试**：保持现有跨 Key 机制。429 默认 3 次，临时错误 2 次，结构化输出 3 次，指数退避并遵循 `Retry-After`。
2. **模型阶段重试**：调用级重试耗尽后写入 Redis 延迟队列，不生成技术性人工复核结果。
3. **worker 恢复**：未处理异常不直接死信。可以写入延迟队列时确认原消息；Redis 不可用时保留 pending 消息，等待 `XAUTOCLAIM` 恢复。
4. **截止时间**：首次提交时间起 30 分钟。截止时间内按配置延迟重试；超过截止时间才写入最终失败并发送失败回调。

## 完整机审边界

- 帧视觉审核失败必须重试，不能生成“模型调用异常”finding。
- 字幕模型审核失败必须重试，不能静默跳过字幕通道。
- 整片剧情与价值观综合失败必须重试，不能只用分段结果假装完整报告。
- 没有字幕、没有违规发现属于合法结果，不属于技术失败。

## 运维与观测

- 新增模型延迟重试队列长度、投递数量、重试次数、错误类型和剩余截止时间事件。
- 保留死信队列，仅用于超过 30 分钟或明确不可恢复的最终失败。
- 部署后先运行故障注入测试，再用历史 198 集重放；最终万分之一指标必须用滚动 10,000 条样本验证。

## 非目标

- 本轮不部署本地视觉模型灾备。
- 本轮不修改审核规则、中文化输出和七牛下载算法；这些作为后续独立改造，避免可靠性修复与审核语义变化互相干扰。
